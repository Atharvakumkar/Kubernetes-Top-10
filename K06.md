# K06: Overly Exposed Kubernetes Components

## Overview:

Kubernetes exposes several components that are responsible for managing the cluster, including the Kubernetes API Server, Kubelet, and etcd. These components are designed for communication between administrators and the cluster.

If these services are directly exposed to the public Internet without proper network restrictions, attackers can discover the cluster, identify its Kubernetes version, inspect TLS certificates, and attempt to authenticate using stolen credentials. In severe cases, exposed Kubelet or etcd services may lead to complete cluster compromise.

## Threat Scenario (OWASP):

A company deploys its Kubernetes cluster on a cloud provider.

During deployment, the Kubernetes API Server is configured with a public IP address and unrestricted access (0.0.0.0/0).

An attacker discovers the API endpoint using Internet scanning tools.

The attacker can now:

Identify the Kubernetes version <br>
Enumerate cluster information <br>
Attempt stolen credentials <br>
Launch brute-force attacks <br>
Exploit known vulnerabilities <br>

Although authentication is enabled, exposing the API significantly increases the attack surface.

## Part A - Vulnerable Kubernetes Configuration:

### Step 1 - Create a Local Kubernetes Cluster

Start Minikube

```
minikube start
```
Verify the cluster

```
kubectl cluster-info
```

You will get the Kubernetes API endpoint.

### Step 2 - Verify API Server is reachable

Command:
```
curl -k https://192.168.49.2:8443/version
```

output:
```
{
    "major":"1",
    "minor":"30",
    "gitVersion":"v1.30.2"
}
```

This API server exposes version information.

### Step 3 - Inspect TLS Certificate

Command:
```
openssl s_client -connect 192.168.49.2:8443
```

The TLS Certificate reveals: <br>
• Internal DNS Names<br>
• Internal IP Addresses<br>
• Cluster Network details<br>

## Part B - Attacker Exploitation: 

### Step 4 - Stimulating a stolen kubeconfig
Assume the attacker obtains an administrator kubeconfig from GitHub

Example: <br>
**clusters:** <br>
<br>
**cluster:** <br>
**--server: https://192.168.49.2:8443**

Now the attacker runs:
``` 
kubectl get pods
```

If the credentials are valid, the attacker gains access to the cluster without needing to be inside the corporate network.

### Step 5 - Enumerate Cluster Resources:
Command:
```
kubectl get nodes

kubectl get namespaces

kubectl get pods -A
```

The attacker can gather: <br>
• Worker node names<br>
• Running applications<br>
• Namespaces<br>
• Cluster topology<br>

### Step 6 - Attempt Secret Enumeration:

Command:
```
kubectl get secrets -A
```

## Observations:

Because the Kubernetes API Server is publically reachable, an attacker can: <br>
• Discover the cluster<br>
• Identify the Kubernetes version<br>
• Inspect the TLS certificates<br>
• Enumerate cluster resources<br>
• Use stolen credentials remotely<br>

## Part C - Applying Security Controls:

### Step 7 - Restrict API Server Access
Instead of allowing:  <br>
**0.0.0.0/0**

Restrict access to: <br>
**Office Network** <br>
**VPN** <br>
**CI/CD Server** <br>
Only trusted hosts should communicate with the Kubernetes API

### Step 8 - Enable RBAC
Verify permissions:
```
kubectl auth can-i get secrets
```

Output:
```
no
```
Users should only receive the permissions required for their job

### Step 9 - Protect Worker Nodes
Ensure Kubelet (10250) 
is accessible only by: <br>
• Kubernetes API Server <br>
• Monitoring Systems <br>   
Never expose it publicly.

### Step 10 - Protect etcd
Ensure port **2379** is accessible only from the control plane. Enable Encryption at Rest for Kubernetes Secrets.

## Part D - Verify the Security Controls

### Test 1 - API Access from unauthorized Network
Attempt
```
kubectl get pods
```

Result - **Connection Timed Out**

The firewall blocks unauthorized access.

### Test 2 - Version Endpoint
Attempt - 
```
curl -k https://API-IP/version
```

Result - 
**Connection refused or Access Denied**
Version information is no longer exposed to untrusted networks.

### Test 3 - Verify RBAC
Command - 
```
kubectl get secrets
```

Result -
**Error from server (Forbidden)**
