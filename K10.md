# K10: Inadequate Logging and Monitoring

## Overview:

Logging and monitoring are critical components of Kubernetes security. Without proper visibility into cluster activities, organizations cannot detect attacks, investigate security incidents, or understand the overall security posture of their Kubernetes environment.

Unlike traditional systems, Kubernetes workloads are highly dynamic. Pods are constantly created, destroyed, and rescheduled across different nodes. Some important security events, such as the creation of ServiceAccount tokens or client certificates, are not permanently stored in the Kubernetes datastore. Therefore, enabling audit logging, centralized log collection, monitoring, and alerting is essential for detecting malicious activity and maintaining a reliable audit trail.

---

# Threat Scenario (OWASP):

A company deploys a Kubernetes cluster in production.

The cluster has several security weaknesses:

- Kubernetes Audit Logging is disabled.
- Worker node logs are not collected.
- Application logs remain only inside Pods.
- No centralized logging solution exists.
- No alerting mechanism is configured.

An attacker compromises an application Pod.

The attacker performs several malicious actions:

- Creates a new ServiceAccount.
- Generates authentication tokens.
- Reads Kubernetes Secrets.
- Deletes evidence after completing the attack.

Because no logging or monitoring solution exists, the security team cannot determine:

- When the attack occurred.
- Which user performed the actions.
- What resources were compromised.
- How the attacker gained access.

# • Deploy Vulnerable Application

### 1 - Verify Audit Logging

Check the API Server configuration.

Command:

```bash
ps -ef | grep kube-apiserver
```

Example Output

```
(No --audit-policy-file configured)
```

### Explanation

The Kubernetes API Server is not configured to generate audit logs.

Sensitive API activities are not recorded.

---

### 2 - Deploy a Sample Application

Create **deployment.yaml**

```yaml
apiVersion: apps/v1
kind: Deployment

metadata:
  name: vulnerable-app

spec:
  replicas: 1

  selector:
    matchLabels:
      app: vulnerable-app

  template:
    metadata:
      labels:
        app: vulnerable-app

    spec:
      containers:

      - name: app
        image: nginx
```

Deploy:

```bash
kubectl apply -f deployment.yaml
```

Verify:

```bash
kubectl get pods
```

Generate a token.

### Explanation

Without audit logging enabled, these security-sensitive events leave no permanent record.

Delete log files.


### Explanation

Local container logs can be deleted by an attacker.

If logs are not forwarded externally, valuable forensic evidence is permanently lost.

# • Understanding Vulnerabilities

### 1 - Enable Kubernetes Audit Logging

Configure the API Server.

Example:

```text
--audit-policy-file=/etc/kubernetes/audit-policy.yaml

--audit-log-path=/var/log/kubernetes/audit.log
```

Restart the API Server.

### Explanation

Every Kubernetes API request is now recorded.

---

### 2 - Configure an Audit Policy

Example **audit-policy.yaml**

```yaml
apiVersion: audit.k8s.io/v1
kind: Policy

rules:

- level: Metadata

- level: RequestResponse

  resources:

  - group: ""
    resources:
    - secrets
```

Apply the configuration by restarting the API Server.

### Explanation

Sensitive operations such as Secret access and authentication events are logged.

---

### 3 - Centralize Cluster Logs

Deploy a centralized logging solution.

Examples:

- Fluent Bit
- Fluentd
- Elasticsearch
- Loki
- OpenSearch
- Splunk

These tools collect logs from:

- Kubernetes API Server
- Worker Nodes
- Containers
- Applications

---

### 4 - Configure Monitoring and Alerting

Deploy monitoring tools such as:

- Prometheus
- Grafana
- Alertmanager

Configure alerts for:

- Unauthorized Secret access.
- New ServiceAccount creation.
- Excessive failed authentication attempts.
- Privileged Pod creation.
- Unexpected API activity.

---

### 5 - Deploy Runtime Security

Install runtime security tools.

Examples:

- Falco
- Tetragon

Detect suspicious events such as:

- Shell execution inside containers.
- Privilege escalation.
- Network scanning.
- Unexpected process execution.

---

### 6 - Prevent Log Loss

Configure persistent log buffering on worker nodes.

Forward logs to an external logging platform using:

- Kafka
- RabbitMQ
- Fluent Bit persistent storage

This ensures logs are retained even if downstream logging services become temporarily unavailable.

# • Fixing the Vulnerability

### 1 - Verify Audit Logs

Create a ServiceAccount.

Command:

```bash
kubectl create serviceaccount test-sa
```

Verify the audit log.

```bash
cat /var/log/kubernetes/audit.log
```

Result

```
User:

admin

Verb:

create

Resource:

serviceaccounts
```

The event is successfully recorded.

---

### 2 - Verify Secret Access Logging

Command:

```bash
kubectl get secrets
```

Verify the audit log.

Result

```
Resource:

secrets

Verb:

get

User:

developer
```

Secret access is now fully auditable.

---

### 3 - Verify Alert Generation

Create multiple Pods.

```bash
kubectl run test1 --image=nginx

kubectl run test2 --image=nginx
```

Result

```
Alert Generated:

Unusual Pod Creation Activity Detected
```

Monitoring systems detect abnormal behavior.

---

### 4 - Verify Centralized Logging

Check the logging platform.

Result

```
Audit Logs

Node Logs

Container Logs

Application Logs
```

Logs from the entire cluster are successfully centralized.

---

## Final Observation

After implementing the recommended logging and monitoring controls:

- Kubernetes Audit Logging records every API request.
- Authentication events and Secret access are permanently logged.
- Cluster, node, and application logs are centralized.
- Monitoring systems detect suspicious behavior in real time.
- Runtime security tools identify malicious activity inside containers.
- Persistent log buffering prevents audit data loss during outages.

As a result, the Kubernetes cluster provides comprehensive visibility into security events, enabling rapid threat detection, effective incident response, and reliable forensic investigation following a security incident.
