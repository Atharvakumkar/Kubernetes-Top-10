# K01: Insecure Workload Configurations

## Overview:

By default, Kubernetes workloads are configured for ease of deployment rather than security. Containers often run with insecure default settings such as running as the root user, receiving unnecessary Linux capabilities, mounting ServiceAccount tokens automatically, and lacking resource limits or seccomp profiles.

If these insecure defaults are not changed, an attacker who compromises an application may be able to escalate privileges, escape the container, access the Kubernetes API, or cause denial-of-service attacks against other workloads.

To improve workload security, Kubernetes provides security controls through the `securityContext` field and Pod specifications, allowing administrators to enforce least privilege and harden workloads.

---

# Threat Scenario (OWASP):

A company deploys a web application on Kubernetes using the default Pod configuration.

The application has several insecure settings:

- Runs as the root user.
- Uses the default Linux capabilities.
- Automatically mounts a ServiceAccount token.
- Has no seccomp profile.
- Has no CPU or memory limits.
- Allows privilege escalation.

An attacker exploits a vulnerability in the application and gains shell access to the container.

The attacker can now:

- Execute commands as the root user.
- Access the Kubernetes API using the mounted ServiceAccount token.
- Attempt container breakout.
- Install malicious software.
- Exhaust node resources and impact other workloads.

Although the application itself is functional, insecure workload configurations significantly increase the impact of a successful compromise.

# Part A - Vulnerable Kubernetes Configuration

### Step 1 - Create a Namespace

Command:

```bash
kubectl create namespace workload-demo
```

Verify:

```bash
kubectl get namespaces
```

---

### Step 2 - Deploy an Insecure Pod

Create **insecure-pod.yaml**

```yaml
apiVersion: v1
kind: Pod

metadata:
  name: insecure-pod
  namespace: workload-demo

spec:
  containers:

  - name: app
    image: nginx
```

Deploy:

```bash
kubectl apply -f insecure-pod.yaml
```

Verify:

```bash
kubectl get pods -n workload-demo
```

### Explanation

The Pod uses Kubernetes default settings with no workload hardening applied.

---

### Step 3 - Verify the Container Runs as Root

Access the Pod.

Command:

```bash
kubectl exec -it insecure-pod -n workload-demo -- sh
```

Inside the container.

```bash
id
```

Output

```
uid=0(root)
gid=0(root)
```

### Explanation

The application is running as the root user, increasing the risk of privilege escalation and container breakout.

---

### Step 4 - Verify the Mounted ServiceAccount Token

Command:

```bash
ls /var/run/secrets/kubernetes.io/serviceaccount
```

Output

```
ca.crt

namespace

token
```

### Explanation

Every Pod automatically receives a ServiceAccount token unless explicitly disabled.

If compromised, an attacker may use this token to communicate with the Kubernetes API.

---

### Step 5 - Verify Resource Limits

Command:

```bash
kubectl describe pod insecure-pod -n workload-demo
```

Output

```
Limits:

None

Requests:

None
```

### Explanation

Without CPU and memory limits, a compromised container can consume excessive node resources and impact other workloads.

Because insecure workload configurations are used, an attacker can:

- Access the Kubernetes API using ServiceAccount tokens.

- Attempt privilege escalation.

# Part C - Applying Security Controls

### Step 10 - Deploy a Hardened Pod

Create **secure-pod.yaml**

```yaml
apiVersion: v1
kind: Pod

metadata:
  name: secure-pod
  namespace: workload-demo

spec:

  automountServiceAccountToken: false

  securityContext:

    runAsUser: 1000
    runAsGroup: 1000

    seccompProfile:
      type: RuntimeDefault

  containers:

  - name: app
    image: nginx

    securityContext:

      allowPrivilegeEscalation: false

      privileged: false

      readOnlyRootFilesystem: true

      capabilities:

        drop:

        - ALL

    resources:

      requests:

        cpu: "100m"
        memory: "128Mi"

      limits:

        cpu: "500m"
        memory: "256Mi"
```

Deploy:

```bash
kubectl apply -f secure-pod.yaml
```

### Explanation

The hardened Pod:

- Runs as a non-root user.
- Drops all Linux capabilities.
- Uses the RuntimeDefault seccomp profile.
- Prevents privilege escalation.
- Uses a read-only root filesystem.
- Does not mount a ServiceAccount token.
- Enforces CPU and memory limits.

---

### Step 11 - Verify Non-Root Execution

Command:

```bash
kubectl exec -it secure-pod -n workload-demo -- id
```

Output

```
uid=1000
gid=1000
```

The application is no longer running as root.

---

### Step 12 - Verify ServiceAccount Token

Command:

```bash
kubectl exec -it secure-pod -n workload-demo -- ls /var/run/secrets/kubernetes.io/serviceaccount
```

Result

```
No such file or directory
```

The ServiceAccount token is no longer mounted.

---

### Step 13 - Verify Read-Only Root Filesystem

Command:

```bash
touch /tmp/test.txt
```

Result

```
Read-only file system
```

Attackers cannot write malicious files to the container filesystem.

---

## Final Observation

After applying the recommended workload security controls:

- Containers run as non-root users.
- Linux capabilities are removed.
- ServiceAccount tokens are mounted only when required.
- Privilege escalation is disabled.
- The RuntimeDefault seccomp profile reduces the attack surface.
- The root filesystem is read-only.
- CPU and memory limits prevent resource exhaustion attacks.

As a result, Kubernetes workloads follow the principle of least privilege, significantly reducing the risk of container breakout, privilege escalation, Kubernetes API abuse, and denial-of-service attacks.
