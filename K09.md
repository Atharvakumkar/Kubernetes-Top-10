# K09: Broken Authentication Mechanisms

## Overview:

Authentication is the first line of defense in a Kubernetes cluster. Every request made to the Kubernetes API Server must be authenticated before authorization policies such as RBAC are evaluated.

Kubernetes supports multiple authentication methods including client certificates, Service Account tokens, OpenID Connect (OIDC), JSON Web Tokens (JWT), and cloud identity providers. While this flexibility allows Kubernetes to integrate with different environments, weak or misconfigured authentication mechanisms can allow attackers to gain unauthorized access, escalate privileges, or compromise the entire cluster.

Organizations should use strong authentication methods, enforce Multi-Factor Authentication (MFA), disable anonymous access, and use short-lived credentials instead of static or long-lived tokens.

---

# Threat Scenario (OWASP):

A company deploys a Kubernetes cluster using the default authentication configuration.

The cluster has several security issues:

- Anonymous authentication is enabled.
- Developers authenticate using long-lived client certificates.
- Pods automatically receive the default ServiceAccount token.
- Long-lived ServiceAccount tokens are used outside the cluster.
- Multi-Factor Authentication (MFA) is not enforced.

An attacker obtains a leaked ServiceAccount token from a public GitHub repository.

The attacker can now:

- Authenticate to the Kubernetes API Server.
- Enumerate cluster resources.
- Access sensitive workloads.
- Escalate privileges using excessive permissions.
- Maintain long-term access because the credentials do not expire.

Although Kubernetes authentication is enabled, weak authentication practices significantly increase the risk of cluster compromise.

---

# • Deploy Vulnerable Application

### 1 - Verify Anonymous Authentication

Check the API Server configuration.

Command:

```bash
ps -ef | grep kube-apiserver
```

Example Output

```text
--anonymous-auth=true
```

### Explanation

Anonymous authentication allows unauthenticated requests to be assigned the `system:anonymous` identity.

If RBAC is misconfigured, attackers may gain unauthorized access.

---

### 2 - Check the Default ServiceAccount

Command:

```bash
kubectl get serviceaccounts
```

Output

```
default
```

Describe the ServiceAccount.

```bash
kubectl describe serviceaccount default
```

### Explanation

Pods automatically receive the default ServiceAccount unless configured otherwise.

This may unintentionally provide Kubernetes API access.

---

### 3 - Verify Automatic Token Mounting

Deploy a simple Pod.

```yaml
apiVersion: v1
kind: Pod

metadata:
  name: vulnerable-pod

spec:
  containers:

  - name: app
    image: nginx
```

Deploy:

```bash
kubectl apply -f vulnerable-pod.yaml
```

Access the Pod.

```bash
kubectl exec -it vulnerable-pod -- sh
```

Check for the mounted token.

```bash
ls /var/run/secrets/kubernetes.io/serviceaccount
```

Output

```
ca.crt

namespace

token
```

### Explanation

The ServiceAccount token is automatically mounted into the Pod.

If the Pod is compromised, the attacker can steal this token.

---

### 4 - Authenticate Using a Long-Lived Token

Suppose an attacker obtains a ServiceAccount token.

Example:

```text
eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...
```

Use the token to access the Kubernetes API.

```bash
kubectl --token=<token> get pods
```

### Explanation

If the token is valid, the attacker can authenticate without requiring a username or password.

---

# • Understanding Vulnerabilities

### 1 - Enumerate Cluster Resources

Command:

```bash
kubectl --token=<token> get namespaces
```

Command:

```bash
kubectl --token=<token> get pods -A
```

### Explanation

The attacker gathers information about the cluster.

---

### 2 - Access Kubernetes Secrets

Command:

```bash
kubectl --token=<token> get secrets -A
```

### Explanation

If excessive RBAC permissions exist, the attacker can retrieve sensitive credentials.

---

### 3 - Verify Anonymous Access

Command:

```bash
curl -k https://<API-SERVER>:6443
```

If anonymous authentication is enabled, the request is processed as:

```
system:anonymous
```

### Explanation

Improper RBAC permissions assigned to anonymous users may expose cluster resources.

---

### 4 - Use a Long-Lived Client Certificate

Suppose a leaked administrator certificate is found.

Command:

```bash
kubectl --client-certificate=admin.crt \
--client-key=admin.key get nodes
```

### Explanation

Since Kubernetes client certificates cannot easily be revoked, attackers may retain long-term access.

---

## Observations:

Because authentication is weak or improperly configured, an attacker can:

- Authenticate using stolen ServiceAccount tokens.
- Abuse long-lived client certificates.
- Exploit anonymous authentication.
- Access Kubernetes resources without MFA.
- Maintain persistent access using static credentials.

---

# • Fixing the Vulnerability

### 1 - Disable Anonymous Authentication

Configure the API Server.

```text
--anonymous-auth=false
```

Restart the API Server after applying the configuration.

### Explanation

Every request must now provide a valid identity.

---

### 2 - Disable Automatic ServiceAccount Token Mounting

Modify the Pod specification.

```yaml
apiVersion: v1
kind: Pod

metadata:
  name: secure-pod

spec:

  automountServiceAccountToken: false

  containers:

  - name: app
    image: nginx
```

Apply:

```bash
kubectl apply -f secure-pod.yaml
```

### Explanation

Pods no longer receive Kubernetes API credentials unless explicitly required.

---

### 3 - Use OIDC Authentication

Configure Kubernetes to authenticate users using an external Identity Provider.

Examples:

- Microsoft Entra ID (Azure AD)
- Google Identity
- Okta
- Keycloak

### Explanation

OIDC provides centralized authentication, user management, auditing, and supports MFA.

---

### 4 - Enforce Multi-Factor Authentication (MFA)

Configure the Identity Provider to require:

- Password
- One-Time Password (OTP)
- Authenticator App
- Security Key

### Explanation

Even if credentials are stolen, attackers cannot authenticate without the second factor.

---

### 5 - Use Short-Lived Tokens

Generate a temporary ServiceAccount token.

Command:

```bash
kubectl create token my-serviceaccount --duration=15m
```

### Explanation

Short-lived credentials automatically expire, reducing the impact of credential theft.

---

### 6 - Create Dedicated ServiceAccounts

Instead of using the default ServiceAccount, create one for each application.

Example:

```yaml
apiVersion: v1
kind: ServiceAccount

metadata:
  name: application-sa
```

Assign only the minimum RBAC permissions required.

---

# • Verifying the Security Controls

### 1 - Verify Anonymous Authentication

Command:

```bash
curl -k https://<API-SERVER>:6443
```

Result

```
401 Unauthorized
```

Anonymous requests are rejected.

---

### 2 - Verify ServiceAccount Token

Access the Pod.

Command:

```bash
kubectl exec -it secure-pod -- sh
```

Check the token directory.

```bash
ls /var/run/secrets/kubernetes.io/serviceaccount
```

Result

```
No such file or directory
```

The ServiceAccount token is no longer automatically mounted.

---

### 3 - Verify Short-Lived Token

Generate a temporary token.

```bash
kubectl create token my-serviceaccount --duration=15m
```

Wait until the token expires.

Attempt:

```bash
kubectl --token=<expired-token> get pods
```

Result

```
Unauthorized
```

Expired credentials cannot be reused.

---

### 4 - Verify OIDC Authentication

Login through the configured Identity Provider.

Result

```
Authentication Successful

MFA Verified
```

Users authenticate using centrally managed identities instead of static credentials.

---

## Final Observation

After implementing the recommended authentication controls:

- Anonymous access to the Kubernetes API is disabled.
- Pods no longer receive unnecessary ServiceAccount tokens.
- Users authenticate through a trusted external Identity Provider using OIDC.
- Multi-Factor Authentication protects user accounts.
- Short-lived tokens replace long-lived static credentials.
- Dedicated ServiceAccounts provide least-privilege access for workloads.

As a result, the Kubernetes cluster is significantly more resilient against credential theft, authentication bypass, privilege escalation, and unauthorized access through compromised identities.
