# K02: Overly Permissive Authorization Configurations

## Overview:

Kubernetes uses Role-Based Access Control (RBAC) to determine what users, applications, and services are allowed to do within a cluster. While RBAC is highly flexible, overly permissive authorization configurations can unintentionally grant users or workloads excessive privileges, increasing the risk of unauthorized access, privilege escalation, and complete cluster compromise.

Common authorization mistakes include assigning the `cluster-admin` role, granting unnecessary permissions through built-in ClusterRoles, allowing access to Kubernetes Secrets, and providing dangerous permissions such as `escalate`, `bind`, or `impersonate`. Following the Principle of Least Privilege is essential to minimize the impact of compromised identities.

---

# Threat Scenario (OWASP):

A company grants developers the built-in `cluster-admin` ClusterRole to simplify cluster management.

Additionally:

- Pods automatically mount ServiceAccount tokens.
- Developers can list Kubernetes Secrets.
- A third-party Helm chart installs an overly permissive ClusterRole.
- Users have permission to impersonate other identities.

An attacker compromises a developer account and gains access to the Kubernetes cluster.

The attacker can now:

- Read Kubernetes Secrets.
- Create privileged workloads.
- Modify RBAC permissions.
- Impersonate administrators.
- Escalate privileges across the cluster.

Although authentication is functioning correctly, excessive authorization permissions allow the attacker to gain full control of the Kubernetes cluster.

# • Deploy Vulnerable Application

### 1 - Create a Namespace

Command:

```bash
kubectl create namespace rbac-demo
```

Verify:

```bash
kubectl get namespaces
```

---

### 2 - Create an Overly Permissive ClusterRoleBinding

Grant a user full administrator access.

Command:

```bash
kubectl create clusterrolebinding developer-admin \
--clusterrole=cluster-admin \
--user=developer
```

Verify:

```bash
kubectl get clusterrolebinding
```

### Explanation

The user **developer** now has unrestricted administrative access to the cluster.

---

### 3 - Verify Permissions

Command:

```bash
kubectl auth can-i --list --as=developer
```

Output (Example)

```
Resources: *

Verbs: *
```

### Explanation

The user can perform any operation on any Kubernetes resource.

---

### 4 - Verify Secret Access

Command:

```bash
kubectl auth can-i list secrets --as=developer
```

Output

```
yes
```

### Explanation

The developer can retrieve Kubernetes Secrets, exposing sensitive credentials.

# • Understanding Vulnerabilities

### 1 - Read Kubernetes Secrets

Command:

```bash
kubectl get secrets -A
```

View a Secret.

```bash
kubectl get secret db-secret -o yaml
```

Decode the Secret.

```bash
echo "<base64-value>" | base64 -d
```

### Explanation

The attacker retrieves sensitive credentials from Kubernetes Secrets.

---

### 2 - Create a Privileged Pod

Command:

```bash
kubectl run attacker \
--image=nginx \
--overrides='
{
  "spec": {
    "containers": [{
      "name":"attacker",
      "image":"nginx",
      "securityContext":{
        "privileged":true
      }
    }]
  }
}'
```

### Explanation

Because the attacker has cluster-admin privileges, privileged workloads can be deployed.

---

### 3 - Impersonate Another User

Command:

```bash
kubectl auth can-i get pods \
--as=system:admin
```

### Explanation

If the attacker has **impersonate** permissions, they may act as another Kubernetes user.

---

### 4 - Escalate Privileges

Create a new ClusterRole with excessive permissions.

Command:

```bash
kubectl create clusterrole attacker-role \
--verb="*" \
--resource="*"
```

Bind the role.

```bash
kubectl create clusterrolebinding attacker-binding \
--clusterrole=attacker-role \
--user=attacker
```

### Explanation

Permissions such as **bind** and **escalate** allow attackers to grant themselves additional privileges.

---

## Observations:

Because authorization is overly permissive, an attacker can:

- Read Kubernetes Secrets.
- Deploy privileged workloads.
- Escalate privileges.
- Impersonate administrators.
- Modify RBAC permissions.
- Gain complete control of the Kubernetes cluster.

---

# • Fixing the Vulnerability

### 1 - Apply the Principle of Least Privilege

Instead of assigning **cluster-admin**, create a Role with only the required permissions.

Example:

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role

metadata:
  name: developer-role
  namespace: rbac-demo

rules:

- apiGroups: [""]

  resources:

  - pods

  verbs:

  - get
  - list
  - watch
```

Apply:

```bash
kubectl apply -f developer-role.yaml
```

### Explanation

Users receive only the permissions necessary to perform their tasks.

---

### 2 - Create a RoleBinding

Bind the Role to the developer.

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding

metadata:
  name: developer-binding
  namespace: rbac-demo

subjects:

- kind: User
  name: developer

roleRef:

  kind: Role
  name: developer-role

  apiGroup: rbac.authorization.k8s.io
```

Apply:

```bash
kubectl apply -f rolebinding.yaml
```

### Explanation

Permissions are now limited to a single namespace and only the required resources.

---

### 3 - Restrict Secret Access

Verify permissions.

Command:

```bash
kubectl auth can-i get secrets --as=developer
```

Result

```
no
```

Users should not receive **get**, **list**, or **watch** permissions on Secrets unless absolutely necessary.

---

### 4 - Review Third-Party RBAC

Inspect installed ClusterRoles.

Command:

```bash
kubectl get clusterroles
```

Review bindings.

```bash
kubectl get clusterrolebindings
```

Ensure third-party applications do not install unnecessary wildcard (`*`) permissions.

---

### 5 - Implement Just-In-Time (JIT) Access

Instead of granting permanent administrator access:

- Issue temporary elevated permissions.
- Automatically revoke RoleBindings after a defined period.
- Use short-lived OIDC tokens.
- Record approvals through GitOps workflows.

### Explanation

JIT Access reduces standing privileges and improves accountability.

---

```

```

Command:

```bash

```

Output

```

```

The developer receives only namespace-scoped permissions.

---

## Final Observation

After implementing proper authorization controls:

- Users receive only the minimum permissions required.
- Cluster-wide administrator privileges are eliminated.
- Access to Kubernetes Secrets is restricted.
- Dangerous permissions such as **bind**, **escalate**, and **impersonate** are tightly controlled.
- Third-party RBAC configurations are reviewed before deployment.
- Just-In-Time (JIT) access replaces permanent privileged accounts with temporary, auditable permissions.

As a result, the Kubernetes cluster follows the Principle of Least Privilege, significantly reducing the risk of privilege escalation, unauthorized access, and full cluster compromise caused by overly permissive authorization configurations.
