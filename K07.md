# K07: Misconfigured and Vulnerable Cluster Components

## Overview:

A Kubernetes cluster consists of several critical components, including the Kubernetes API Server, Controller Manager, Scheduler, Kubelet, Kube-Proxy, container runtime, and worker nodes. If these components are misconfigured or running outdated software, attackers can exploit known vulnerabilities or insecure settings to compromise workloads and even gain control of the entire cluster.

By default, Kubernetes components are designed for flexibility rather than security. Without proper hardening, regular patching, and secure configuration, attackers can exploit exposed services, weak authentication, excessive permissions, or outdated software versions.

Organizations should follow security best practices such as the CIS Kubernetes Benchmark, use hardened operating systems, and keep all cluster components updated to reduce the attack surface.

---

# Threat Scenario (OWASP):

A company deploys a Kubernetes cluster using default settings.

The cluster has several security issues:

- Kubelet is running with insecure configuration.
- Worker nodes use a general-purpose Linux distribution.
- Kubernetes components have not been updated for several months.
- Anonymous authentication is enabled on Kubelet.
- The container runtime is running an outdated version.

An attacker scans the cluster and discovers vulnerable Kubernetes components.

The attacker can now:

- Exploit known Kubernetes vulnerabilities.
- Access insecure Kubelet APIs.
- Escalate privileges on worker nodes.
- Compromise the container runtime.
- Gain control of workloads running in the cluster.

Although RBAC and authentication are configured, outdated and misconfigured cluster components provide attackers with an easy path to compromise the cluster.

# • Deploy Vulnerable Application

### 1 - Check Kubernetes Version

Command:

```bash
kubectl version
```

Output

```text
Server Version: v1.25.x
```

### Explanation

Older Kubernetes versions may contain publicly known security vulnerabilities that have already been patched in newer releases.

---

### 2 - Inspect Kubelet Configuration

Command:

```bash
ps -ef | grep kubelet
```

Example Output

```text
kubelet \
--anonymous-auth=true \
--read-only-port=10255
```

### Explanation

The Kubelet is configured with insecure settings:

- Anonymous authentication enabled.
- Read-only port exposed.

These settings increase the attack surface.

---

### 3 - Check Node Operating System

Command:

```bash
kubectl get nodes -o wide
```

Access a worker node.

```bash
cat /etc/os-release
```

Output

```text
Ubuntu 20.04
```

### Explanation

A general-purpose operating system may include unnecessary packages and services that increase the attack surface.

---

### 4 - Check Container Runtime Version

Command:

```bash
crictl version
```

Output

```text
containerd 1.5.x
```

### Explanation

Older container runtime versions may contain publicly disclosed vulnerabilities.

# • Understanding Vulnerabilities

### 1 - Identify Kubernetes Version

Command:

```bash
kubectl version
```

The attacker identifies the Kubernetes version and searches for publicly known vulnerabilities affecting that release.

---

### 2 - Access the Kubelet API

Command:

```bash
curl http://<node-ip>:10255/pods
```

Result

```text
Pod information displayed
```

### Explanation

Because the read-only Kubelet port is enabled, an attacker can gather information about running workloads.

---

### 3 - Exploit Outdated Software

The attacker identifies that the Kubernetes version or container runtime is affected by a known CVE.

Using publicly available exploit code, the attacker targets the vulnerable component and gains elevated privileges.

### Explanation

Running outdated software significantly increases the risk of successful attacks.

---

### 4 - Enumerate Worker Nodes

Command:

```bash
kubectl get nodes
```

The attacker identifies worker nodes and targets them for further compromise.

---

## Observations:

Because cluster components are outdated or insecurely configured, an attacker can:

- Exploit known Kubernetes vulnerabilities.
- Access insecure Kubelet APIs.
- Gather information about cluster workloads.
- Target vulnerable worker nodes.
- Escalate privileges using publicly known exploits.

# • Fixing the Vulnerability

### 1 - Upgrade Kubernetes

Verify the current version.

Command:

```bash
kubectl version
```

Upgrade the cluster to a supported Kubernetes release following the official upgrade procedure provided by your Kubernetes distribution.

### Explanation

Keeping Kubernetes up to date ensures known security vulnerabilities are patched.

---

### 2 - Harden Kubelet Configuration

Disable anonymous authentication.

Example configuration:

```text
--anonymous-auth=false
```

Disable the read-only port.

```text
--read-only-port=0
```

Restart the Kubelet service after applying the changes.

### Explanation

Only authenticated users should be able to communicate with the Kubelet.

---

### 3 - Harden Worker Nodes

Use a container-optimized operating system such as:

- Bottlerocket (AWS)
- Container-Optimized OS (Google Cloud)
- Flatcar Linux
- Talos Linux

These operating systems minimize unnecessary packages and services.

---

### 4 - Follow the CIS Kubernetes Benchmark

Use tools such as:

- Kubescape
- kube-bench

Example:

```bash
kubescape scan framework nsa
```

or

```bash
kube-bench
```

These tools automatically identify security misconfigurations based on industry best practices.

---

### 5 - Keep Container Runtime Updated

Verify the runtime version.

Command:

```bash
crictl version
```

Update the container runtime to the latest supported release.

This helps protect against vulnerabilities in containerd or CRI-O.

## Final Observation

After hardening and updating the Kubernetes cluster:

- Kubernetes components are running supported versions.
- Kubelet insecure settings have been disabled.
- Worker nodes use hardened operating systems.
- Container runtimes are updated and securely configured.
- CIS Kubernetes Benchmark recommendations have been applied.
- Continuous security scanning helps identify future misconfigurations.

As a result, the Kubernetes cluster is significantly more resistant to attacks targeting vulnerable or misconfigured components, reducing the likelihood of privilege escalation, remote compromise, and exploitation of known security vulnerabilities.
