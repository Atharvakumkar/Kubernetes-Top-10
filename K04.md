# K04: Lack of Cluster Level Policy Enforcement

## Overview:

By default, Kubernetes provides significant flexibility in how workloads are deployed and managed. While this flexibility enables rapid application development, it does not enforce security policies across the entire cluster. As a result, users with sufficient permissions can deploy workloads that violate organizational security standards.

Without centralized policy enforcement, workloads may run as privileged containers, use untrusted container images, expose sensitive services, or bypass security controls. As organizations scale, inconsistent configurations across multiple teams increase the risk of security incidents.

Kubernetes provides several built-in policy enforcement mechanisms such as **Pod Security Admission (PSA)**, **Validating Admission Policies (VAP)**, and **Admission Webhooks** to ensure that workloads comply with organizational security requirements before they are deployed.

---

# Threat Scenario (OWASP):

A company operates a Kubernetes cluster shared by multiple development teams.

Developers have permission to deploy applications directly into the cluster.

Because no cluster-wide security policies are enforced:

- A developer deploys a privileged container.
- Another team pulls images from an untrusted Docker registry.
- A Pod runs as the root user.
- Sensitive namespaces have no Pod Security policies.
- Internal applications communicate freely due to unrestricted networking.

An attacker compromises one vulnerable application.

The attacker can now:

- Deploy privileged containers.
- Escape container isolation.
- Pull malicious container images.
- Move laterally across the cluster.
- Access sensitive workloads.

Although Kubernetes authentication and RBAC are configured correctly, the absence of cluster-wide policy enforcement allows insecure workloads to be deployed.

# • Deploy Vulnerable Application

### 1 - Create a Namespace

Command:

```bash
kubectl create namespace insecure-demo
```

Verify:

```bash
kubectl get namespaces
```

---

### 2 - Deploy a Privileged Pod

Create **privileged-pod.yaml**

```yaml
apiVersion: v1
kind: Pod

metadata:
  name: privileged-pod
  namespace: insecure-demo

spec:
  containers:
  - name: nginx
    image: nginx

    securityContext:
      privileged: true
```

Deploy the Pod.

Command:

```bash
kubectl apply -f privileged-pod.yaml
```

Verify:

```bash
kubectl get pods -n insecure-demo
```

### Explanation

Because no policy exists, Kubernetes allows a privileged container to run.

Privileged containers receive almost unrestricted access to the host operating system.

---

### 3 - Deploy an Image from an Untrusted Registry

Create **untrusted-image.yaml**

```yaml
apiVersion: v1
kind: Pod

metadata:
  name: test-image
  namespace: insecure-demo

spec:
  containers:

  - name: app
    image: randomuser/myimage:latest
```

Deploy:

```bash
kubectl apply -f untrusted-image.yaml
```

### Explanation

Without admission policies, Kubernetes accepts container images from any registry.

---

### 4 - Deploy a Pod Running as Root

Create **root-user.yaml**

```yaml
apiVersion: v1
kind: Pod

metadata:
  name: root-pod
  namespace: insecure-demo

spec:
  containers:

  - name: app
    image: nginx
```

Deploy:

```bash
kubectl apply -f root-user.yaml
```

### Explanation

Since no security policy exists, the Pod runs using the container's default user, which is often the root user.

# • Understanding Vulnerabilities

### 1 - Verify Privileged Container

Command:

```bash
kubectl describe pod privileged-pod -n insecure-demo
```

Output

```
Privileged: true
```

### Explanation

The attacker now controls a privileged container with elevated access to the host.

---

### 2 - Verify Root User

Access the container.

```bash
kubectl exec -it root-pod -n insecure-demo -- sh
```

Inside the container

```bash
id
```

Output

```
uid=0(root)
```

### Explanation

Running containers as the root user increases the impact of container compromise.

---

### 3 - Verify Image Source

Command:

```bash
kubectl describe pod test-image -n insecure-demo
```

Output

```
Image:

randomuser/myimage:latest
```

### Explanation

The cluster allows images from arbitrary registries without validation.

---

## Observations:

Because no cluster-level policy enforcement exists, an attacker can:

- Deploy privileged containers.
- Run workloads as the root user.
- Pull images from untrusted registries.
- Bypass organizational security standards.
- Increase the likelihood of privilege escalation and lateral movement.

---

# • Fixing the Vulnerability

### 1 - Enable Pod Security Admission (PSA)

Apply namespace labels.

Command:

```bash
kubectl label namespace insecure-demo \
pod-security.kubernetes.io/enforce=baseline \
pod-security.kubernetes.io/audit=baseline \
pod-security.kubernetes.io/warn=baseline
```

Verify:

```bash
kubectl get namespace insecure-demo --show-labels
```

### Explanation

Pod Security Admission blocks workloads that violate the selected Pod Security Standard.

---

### 2 - Create a Validating Admission Policy

Example:

```yaml
apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingAdmissionPolicy

metadata:
  name: require-non-root

spec:

  validations:

  - expression: "object.spec.containers.all(c, has(c.securityContext) && has(c.securityContext.runAsNonRoot) && c.securityContext.runAsNonRoot)"
    message: "Containers must run as non-root users."
```

Apply:

```bash
kubectl apply -f vap.yaml
```

### Explanation

The policy rejects any Pod that does not run as a non-root user.

---

### 3 - Restrict Container Registries

Example using Kyverno:

```yaml
validate:

  message: "Only approved registries are allowed."

  pattern:

    spec:

      containers:

      - image: "docker.io/company/*"
```

Only trusted container registries are permitted.

---

### 4 - Enable Admission Webhooks

Deploy a policy engine such as:

- Kyverno
- OPA Gatekeeper
- Kubewarden

These tools enforce custom organizational security policies before resources are created.

---

### 5 - Enable Audit Mode

Instead of blocking workloads immediately, start in audit mode.

Example:

```
pod-security.kubernetes.io/audit=baseline
```

Policy violations are logged without disrupting existing applications.

---

# • Verifying the Security Controls

### 1 - Deploy Privileged Pod

Attempt

```bash
kubectl apply -f privileged-pod.yaml
```

Result

```
Error from server (Forbidden)
```

The Pod Security Admission controller rejects the workload.

---

### 2 - Deploy Root Container

Attempt

```bash
kubectl apply -f root-user.yaml
```

Result

```
Validation failed:

Containers must run as non-root users.
```

The Validating Admission Policy blocks the deployment.

---

### 3 - Deploy Image from Untrusted Registry

Attempt

```bash
kubectl apply -f untrusted-image.yaml
```

Result

```
Admission webhook denied the request.
```

The policy engine rejects the container image.

---

### 4 - Verify Namespace Labels

Command

```bash
kubectl get namespace insecure-demo --show-labels
```

Output

```
pod-security.kubernetes.io/enforce=baseline

pod-security.kubernetes.io/audit=baseline

pod-security.kubernetes.io/warn=baseline
```

The namespace is protected by Pod Security Admission.

---

## Final Observation

After implementing cluster-level policy enforcement:

- Privileged containers are blocked.
- Containers must run as non-root users.
- Images can only be pulled from trusted registries.
- Pod Security Standards are consistently enforced.
- Admission Controllers validate every workload before deployment.
- Policy violations are logged and audited for compliance.

As a result, the Kubernetes cluster maintains a consistent security posture, prevents insecure workload deployments, and significantly reduces the risk of privilege escalation, misconfigurations, and lateral movement.
