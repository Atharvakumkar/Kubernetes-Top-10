# K05: Missing Network Segmentation Controls

## Overview:

By default, Kubernetes allows unrestricted communication between Pods within the cluster. Any Pod can communicate with any other Pod unless Network Policies are explicitly configured. While this simplifies application deployment and service communication, it also creates a flat network architecture that attackers can exploit.

If an attacker compromises a single Pod, they may be able to scan the internal network, communicate with sensitive services, access the Kubernetes API Server, or exfiltrate data to external systems. Proper network segmentation using Kubernetes Network Policies is essential to limit lateral movement and reduce the impact of a compromised workload.

---

# Threat Scenario (OWASP):

A company deploys several applications into a Kubernetes cluster.

The cluster contains:

- Frontend Application
- Backend API
- Database
- Internal Monitoring Service

No Network Policies are configured.

An attacker exploits a vulnerability in the Frontend application and gains access to its Pod.

The attacker can now:

- Scan the internal cluster network.
- Connect to backend services.
- Access internal databases.
- Reach the Kubernetes API Server.
- Exfiltrate sensitive data to external servers.

Although only one application was compromised, the lack of network segmentation allows the attacker to move freely throughout the cluster.

# Part A - Vulnerable Kubernetes Configuration

### Step 1 - Create a Namespace

Command:

```bash
kubectl create namespace network-demo
```

Verify:

```bash
kubectl get namespaces
```

---

### Step 2 - Deploy Frontend and Backend Applications

Create **frontend.yaml**

```yaml
apiVersion: apps/v1
kind: Deployment

metadata:
  name: frontend
  namespace: network-demo

spec:
  replicas: 1

  selector:
    matchLabels:
      app: frontend

  template:
    metadata:
      labels:
        app: frontend

    spec:
      containers:
      - name: frontend
        image: nginx
```

Deploy:

```bash
kubectl apply -f frontend.yaml
```

Create **backend.yaml**

```yaml
apiVersion: apps/v1
kind: Deployment

metadata:
  name: backend
  namespace: network-demo

spec:
  replicas: 1

  selector:
    matchLabels:
      app: backend

  template:
    metadata:
      labels:
        app: backend

    spec:
      containers:
      - name: backend
        image: nginx
```

Deploy:

```bash
kubectl apply -f backend.yaml
```

Verify:

```bash
kubectl get pods -n network-demo
```

### Explanation

No Network Policies exist.

All Pods can communicate with one another by default.

---

### Step 3 - Verify Connectivity Between Pods

Get the frontend Pod.

```bash
kubectl get pods -n network-demo
```

Access the frontend container.

```bash
kubectl exec -it <frontend-pod> -n network-demo -- sh
```

Install curl (if required).

```bash
apt update

apt install curl -y
```

Access the backend Pod.

```bash
curl http://backend
```

### Explanation

The request succeeds because Kubernetes allows unrestricted Pod-to-Pod communication.

# Part C - Applying Security Controls

### Step 8 - Create a Default Deny Network Policy

Create **default-deny.yaml**

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy

metadata:
  name: default-deny-all
  namespace: network-demo

spec:
  podSelector: {}

  policyTypes:
  - Ingress
  - Egress
```

Apply the policy.

Command:

```bash
kubectl apply -f default-deny.yaml
```

Verify:

```bash
kubectl get networkpolicy -n network-demo
```

### Explanation

The policy blocks all inbound and outbound traffic within the namespace.

---

### Step 9 - Allow Only Required Traffic

Create **allow-frontend-backend.yaml**

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy

metadata:
  name: allow-frontend

  namespace: network-demo

spec:

  podSelector:

    matchLabels:
      app: backend

  ingress:

  - from:

    - podSelector:

        matchLabels:
          app: frontend
```

Apply:

```bash
kubectl apply -f allow-frontend-backend.yaml
```

### Explanation

Only the frontend application is allowed to communicate with the backend service.

---

### Step 10 - Restrict Outbound Internet Access

Example:

```yaml
policyTypes:

- Egress
```

Allow outbound traffic only to:

- DNS
- Required APIs
- Approved external services

All other outbound traffic is denied.

---

### Step 11 - Avoid Host Networking

Ensure Pods do not use:

```yaml
hostNetwork: true
```

Unless absolutely required.

Using host networking bypasses normal container network isolation.

---

### Step 12 - Verify CNI Supports Network Policies

Verify the installed CNI.

Command:

```bash
kubectl get pods -n kube-system
```

Use a CNI that supports Network Policies such as:

- Calico
- Cilium
- Antrea

Avoid using CNIs that ignore Network Policies, such as older Flannel-only configurations.

---

## Final Observation

After implementing Kubernetes Network Policies:

- Pod-to-Pod communication follows the principle of least privilege.
- Unauthorized lateral movement is prevented.
- Outbound traffic is restricted to approved destinations.
- Kubernetes control plane services are protected from compromised workloads.
- Sensitive applications are isolated using namespace-level network segmentation.
- The overall attack surface of the Kubernetes cluster is significantly reduced.

As a result, compromising a single Pod no longer allows an attacker to freely explore, communicate with, or exfiltrate data from the rest of the Kubernetes environment.
