# K03: Secrets Management Failures

## Overview:

Applications running inside Kubernetes clusters require access to sensitive information such as database passwords, API keys, tokens, and certificates. If these secrets are not managed securely, attackers who compromise the cluster or gain unauthorized access can retrieve these credentials and use them to access critical infrastructure, cloud services, or databases.

Common mistakes include storing secrets inside container images, ConfigMaps, source code repositories, environment variables, or exposing them through application logs. Kubernetes also stores Secret objects inside etcd, and without proper encryption and access controls, attackers may retrieve these secrets.

---

## Threat Scenario (OWASP):

A company deploys a web application on Kubernetes.

To simplify deployment, developers:

- Store database credentials inside a ConfigMap.
- Commit API keys to a GitHub repository.
- Pass secrets as environment variables.
- Grant developers LIST permission on Secret objects.

An attacker compromises a developer account and gains limited access to the Kubernetes cluster.

The attacker can now:

- Read Kubernetes Secrets.
- View sensitive environment variables.
- Retrieve credentials from ConfigMaps.
- Extract secrets from application logs.
- Access cloud services and databases using stolen credentials.

Although the application itself is secure, improper secrets management allows the attacker to compromise critical resources.

# • Deploy Vulnerable Application

### 1 - Create a Namespace

Command:

```bash
kubectl create namespace secrets-demo
```

---

### 2 - Create an Insecure ConfigMap

Create a file named **configmap.yaml**

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: secrets-demo
data:
  DB_USERNAME: admin
  DB_PASSWORD: SuperSecret123
```

Apply the ConfigMap.

Command:

```bash
kubectl apply -f configmap.yaml
```

Verify:

```bash
kubectl get configmap -n secrets-demo
```

### Explanation

The database password is stored inside a ConfigMap.

ConfigMaps are intended for non-sensitive configuration and should never contain passwords or API keys.

---

### 3 - Deploy an Application Using Environment Variables

Create **deployment.yaml**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: vulnerable-app
  namespace: secrets-demo

spec:
  replicas: 1

  selector:
    matchLabels:
      app: vulnerable-app

  template:
    metadata:
      labels:
        app: vulnerable-app

    spec:
      containers:

      - name: app
        image: nginx

        env:

        - name: DB_USERNAME
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: DB_USERNAME

        - name: DB_PASSWORD
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: DB_PASSWORD
```

Apply the deployment.

Command:

```bash
kubectl apply -f deployment.yaml
```

Verify:

```bash
kubectl get pods -n secrets-demo
```

### Explanation

The application loads sensitive credentials from a ConfigMap and exposes them as environment variables.

# • Understanding Vulnerabilities

### 1 - Read the ConfigMap

Command:

```bash
kubectl get configmap app-config -n secrets-demo -o yaml
```

Output:

```yaml
data:
  DB_USERNAME: admin
  DB_PASSWORD: SuperSecret123
```

The attacker immediately retrieves the database credentials.

---

### 2 - Read Environment Variables

Find the Pod name.

Command:

```bash
kubectl get pods -n secrets-demo
```

Access the container.

```bash
kubectl exec -it <pod-name> -n secrets-demo -- sh
```

Inside the container:

```bash
env
```

Output:

```
DB_USERNAME=admin

DB_PASSWORD=SuperSecret123
```

### Explanation

Environment variables expose sensitive information to anyone who gains access to the container.

---

### 3 - Read Kubernetes Secrets

Suppose the attacker has LIST permission.

Command:

```bash
kubectl get secrets -n secrets-demo
```

View the Secret.

```bash
kubectl get secret db-secret -n secrets-demo -o yaml
```

Decode the password.

```bash
echo "U3VwZXJTZWNyZXQxMjM=" | base64 -d
```

Output

```
SuperSecret123
```

### Explanation

Kubernetes Secrets are Base64 encoded, not encrypted.

Anyone with permission can retrieve and decode them.

---

### 4 - Retrieve Secrets from Logs

Command:

```bash
kubectl logs <pod-name> -n secrets-demo
```

Example Output

```
Connecting to database...

Username=admin

Password=SuperSecret123
```

### Explanation

If developers log configuration values during startup, secrets become visible in application logs.

---

## Observations:

Because secrets are managed insecurely, an attacker can:

- Read credentials stored in ConfigMaps.
- View sensitive environment variables.
- Decode Kubernetes Secrets.
- Retrieve secrets from application logs.
- Gain unauthorized access to databases and cloud resources.

---

# • Fixing the Vulnerability

### 1 - Store Credentials in Kubernetes Secrets

Create **secret.yaml**

Command:

```bash
kubectl create secret generic db-secret \
--from-literal=username=admin \
--from-literal=password=SuperSecret123 \
-n secrets-demo
```

Verify:

```bash
kubectl get secrets -n secrets-demo
```

---

### 2 - Mount Secrets as Files

Instead of using environment variables, mount the Secret as a volume.

Example:

```yaml
volumes:

- name: secret-volume

  secret:
    secretName: db-secret
```

The application now reads the credentials directly from mounted files.

---

### 3 - Restrict Secret Access Using RBAC

Verify permissions.

Command:

```bash
kubectl auth can-i get secrets
```

Output

```
no
```

Only authorized users should have access to Secret resources.

---

### 4 - Enable Encryption at Rest

Configure the Kubernetes API Server with an encryption provider configuration.

Example API Server option:

```text
--encryption-provider-config=/etc/kubernetes/encryption-config.yaml
```

This encrypts Secret objects before storing them in etcd.

---

### 5 - Use External Secret Management

Instead of storing long-lived credentials inside Kubernetes:

Use

- External Secrets Operator
- HashiCorp Vault
- AWS Secrets Manager
- Azure Key Vault
- Google Secret Manager

Applications retrieve secrets dynamically using Workload Identity instead of static passwords.

---

### 6 - Rotate Secrets Regularly

Use

- External Secrets Operator
- cert-manager
- TokenRequest API

This ensures credentials are automatically rotated and short-lived.

---

# • Verifying the Security Controls

### 1 - Verify RBAC

Command:

```bash
kubectl get secrets
```

Result

```
Error from server (Forbidden)
```

Unauthorized users cannot access Secret objects.

---

### 2 - Verify Mounted Secrets

Command:

```bash
ls /etc/secrets
```

Output

```
username

password
```

The application reads secrets from files instead of environment variables.

---

### 3 - Verify Environment Variables

Command:

```bash
env
```

Result

```
(No sensitive credentials displayed)
```

Secrets are no longer exposed through environment variables.

---

### 4 - Verify Application Logs

Command:

```bash
kubectl logs <pod-name>
```

Result

```
Application started successfully.
```

Sensitive credentials no longer appear in application logs.

---

## Final Observation

After applying the recommended security controls:

- Sensitive information is stored in Kubernetes Secret objects or an external secret manager.
- Secrets are mounted as files instead of environment variables.
- RBAC prevents unauthorized users from accessing Secret resources.
- Secret data is encrypted before being stored in etcd.
- Static credentials are replaced with short-lived identities where possible.
- Secrets are automatically rotated to reduce the impact of credential theft.

As a result, the Kubernetes cluster is significantly more resilient against credential theft, unauthorized access, and accidental secret disclosure.
